Password Entropy Explained: Why Randomness Is the Only Thing That Matters

July 2025 7 min read By Zecure Team

Password strength isn't about having a capital letter, a number, and a symbol. It's about entropy — a measure of unpredictability. A password can look complex and be trivially guessable. Here's why randomness is the only thing that matters.

What Is Entropy, Exactly?

In information theory, entropy measures how much uncertainty is in a random variable. Applied to passwords, it answers: how many guesses would an attacker need, on average, to crack this?

Entropy is measured in bits. Each additional bit doubles the search space. A password with 40 bits of entropy has 240 (~1 trillion) possible values. A password with 80 bits has 280 — about a trillion times more.

The formula is simple:

entropy (bits) = log₂(character_set_size ^ password_length)
               = password_length × log₂(character_set_size)

So a truly random 12-character password using 94 printable ASCII characters has:

12 × log₂(94) ≈ 12 × 6.55 ≈ 78.6 bits

The key word is truly random. The formula only holds if each character was chosen independently at random from the full set. Passwords based on words, patterns, or keyboard walks have far less entropy than their length suggests.

Why "Complex" Passwords Fail

Most people, when asked to create a "complex" password, do something like this: take a word, capitalize the first letter, swap some letters for numbers, add a symbol at the end. Summer@2024. P@ssw0rd!. Football1!.

Attackers know this. Password cracking tools don't try random character combinations — they use rule-based attacks: dictionary words with common substitutions, appended years, predictable symbols. Summer@2024 is in most cracking rulesets and falls in seconds.

The Character Set Multiplier

Entropy scales with the size of your character set. Each character you add to the available pool increases entropy per character:

Character set Pool size Bits per character
Digits only (0–9)103.32 bits
Lowercase letters264.70 bits
Lower + uppercase525.70 bits
Alphanumeric625.95 bits
Full printable ASCII946.55 bits

Adding symbols matters — but length matters more. An 8-character password with symbols (52 bits) is weaker than a 12-character lowercase-only password (56 bits), assuming true randomness in both cases.

Against MD5 (still common in old systems), a 40-bit password would fall in under an hour. Against bcrypt, a 60-bit password might take years — but bcrypt requires the service to actually use it. You can't control how a site stores your password.

Design for the worst case. Assume every site you use stores passwords with MD5 and no salt. Then aim for passwords with at least 80 bits of entropy — that gives you ~1024 guesses before cracking, which is computationally infeasible even with massive hardware.

How Many Bits Is Enough?

  • <40 bits — Crackable online or offline in seconds to hours. Not acceptable.
  • 40–60 bits — Crackable offline in hours to months with dedicated hardware. Marginal.
  • 60–80 bits — Offline cracking takes years against weak hash functions. Acceptable with good hashing.
  • 80–128 bits — Computationally infeasible to crack regardless of hashing. Target this range.
  • >128 bits — Used for cryptographic keys, not passwords. Overkill for logins, right-sized for secrets.

For a login password, a randomly generated 16-character alphanumeric password gives ~95 bits. For an API key or encryption secret, 32 random bytes (256 bits) is standard.

Passphrases vs. Random Characters

The XKCD "correct horse battery staple" approach works — but only with truly random word selection from a large wordlist. The Diceware wordlist has ~7,776 words (the BIP-39 list has 2,048). Four random Diceware words gives:

4 × log₂(7776) ≈ 4 × 12.9 ≈ 51.7 bits    // borderline
6 × log₂(7776) ≈ 6 × 12.9 ≈ 77.5 bits    // good

Six random Diceware words is strong and memorable. But "correct horse battery staple" with words you chose yourself? Those words aren't random — they're whatever came to mind first, which dramatically lowers the actual entropy.

Practical Takeaways

  1. Use a password manager to generate and store truly random passwords — you don't need to remember them
  2. Aim for at least 16 random characters using the full ASCII set (≈105 bits) for online accounts
  3. For API keys and secrets, use 32 random bytes (256 bits) — entropy to spare
  4. Never base a password on a real word, name, date, or keyboard pattern
  5. Complexity rules (must have uppercase, number, symbol) are a poor proxy for entropy — a truly random 16-char lowercase password beats any 8-char "complex" password
