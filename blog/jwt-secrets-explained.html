What Is a JWT Secret and How Does It Work?

April 2025 9 min read By Zecure Team

JWT tokens are used to authenticate users in millions of web apps. The entire security model collapses if the signing secret is weak or leaked. Here's exactly how JWTs work — and what a proper secret looks like.

What Is a JSON Web Token?

A JSON Web Token (JWT) is a compact, URL-safe way to represent claims between two parties. In practice, they're used almost exclusively for authentication: after you log in, the server hands you a JWT. Every subsequent request includes that token, and the server verifies it without hitting a database.

The key property that makes this work: the server can verify a JWT it issued without storing any session state. All the information needed for verification is baked into the token itself — and the signature proves it hasn't been tampered with.

Anatomy of a JWT

A JWT is three Base64URL-encoded strings joined by dots:

Header
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
Payload
eyJzdWIiOiJ1c3IxMjMiLCJpYXQiOjE3MTM4MDAwMDB9
.
Signature
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

1. Header

The header declares the token type and the signing algorithm. Decoded, it looks like:

{ "alg": "HS256", "typ": "JWT" }

2. Payload

The payload contains claims — statements about the user and the token itself. Standard claims include sub (subject / user ID), iat (issued at), and exp (expiry). You can add any custom fields:

{
  "sub": "usr_8f3kx29",
  "role": "admin",
  "iat": 1713800000,
  "exp": 1713886400
}

The payload is not encrypted — only signed. Anyone who intercepts a JWT can base64-decode it and read all the claims. Never put passwords, PII, or sensitive data in a JWT payload unless you use JWE (JSON Web Encryption) instead.

3. Signature

The signature is the only part that provides security. For HS256 (HMAC-SHA256), it's computed as:

HMAC-SHA256(
  base64url(header) + "." + base64url(payload),
  secret
)

The server verifies incoming tokens by recomputing this signature and checking it matches. If an attacker modifies any byte of the header or payload, the signature check fails — the token is rejected.

Why the Secret Is Everything

The entire model above depends on one assumption: only your server knows the secret. If an attacker learns the secret, they can forge arbitrary tokens — claiming to be any user, with any role, for any expiry. There's no mechanism to detect this. Your application will happily accept forged admin tokens.

Real incident: In 2022, researchers found several popular apps using "secret", "changeme", or project names as JWT secrets. Attackers can brute-force weak secrets offline using tools like hashcat against intercepted tokens — no rate limiting applies because verification is local.

What Makes a Good JWT Secret?

The secret must be:

  • Long enough — at least 256 bits (32 bytes) for HS256; 512 bits (64 bytes) for HS512
  • Cryptographically random — generated by a CSPRNG, not derived from a phrase or a hash of a password
  • Unique per application — never share secrets between environments or services
  • Kept secret — stored in an environment variable or secrets manager, never in source code or config files

Generating a Secure JWT Secret

// Node.js
const { randomBytes } = require('crypto');
const secret = randomBytes(64).toString('base64');
// → 88-char base64 string, 512 bits of randomness

// Python
import secrets
secret = secrets.token_urlsafe(64)
# → 86-char URL-safe base64 string

// Browser (Web Crypto API)
const bytes = new Uint8Array(64);
crypto.getRandomValues(bytes);
const secret = btoa(String.fromCharCode(...bytes));

HS256 vs. RS256: Symmetric vs. Asymmetric

HS256 uses a single shared secret — the same key signs and verifies. RS256 uses an RSA key pair: the private key signs tokens, and the public key verifies them. This matters when multiple services need to verify tokens:

  • HS256 — simple, fast. Good when a single backend service issues and verifies tokens. Anyone who can verify tokens can also forge them.
  • RS256 / ES256 — better for microservices. You can distribute the public key freely; only the auth service holding the private key can mint new tokens.

For a single-service application, HS256 with a strong secret is perfectly fine. For distributed systems where multiple services verify tokens, prefer RS256 or ES256.

Common JWT Mistakes to Avoid

1. The "none" algorithm attack

Some older JWT libraries accept "alg": "none" and skip signature verification entirely. An attacker can craft a token with no signature and gain access if the library isn't configured to reject unsigned tokens. Always specify the accepted algorithms explicitly.

// Express / jsonwebtoken — always pin the algorithm
jwt.verify(token, secret, { algorithms: ['HS256'] });

2. No expiry (exp)

JWTs with no expiry last forever. A stolen token cannot be invalidated without server-side blocklisting — defeating the stateless design. Always set a short expiry (15 minutes to 24 hours depending on use case) and use refresh tokens for long sessions.

3. Storing tokens in localStorage

Any JavaScript running on your page can read localStorage. An XSS vulnerability is all it takes to steal every user's token. Store access tokens in httpOnly cookies — JavaScript can't read them, but they're sent automatically with each request.

Summary

JWTs are stateless, verifiable, and widely supported — but the security guarantee is entirely dependent on the signing secret. Use a randomly generated secret of at least 32 bytes, store it in an environment variable, and rotate it if you suspect compromise. The algorithm is public; the secret is the only thing standing between your application and token forgery.
