API Keys vs. Passwords: What's the Difference?

February 2025 6 min read By Zecure Team

API keys aren't passwords — and treating them the same is one of the most common security mistakes developers make. They have different formats, different storage requirements, different rotation strategies, and different threat models.

The Short Answer

A password authenticates a human to a system. It's typed, remembered, and hashed server-side before storage. A API key authenticates a machine (your code) to a service. It's never typed, never memorized, and should never be hashed — it needs to be stored and transmitted as-is.

How API Keys Are Generated

A well-formed API key is simply a long string of cryptographically random bytes, usually encoded as hex or base64. The "sk-" or "pk-" prefixes many services use are just conventions to help users identify the key type — they're not part of the security.

// Node.js — generate an API key in the style of OpenAI/Stripe
const { randomBytes } = require('crypto');

function generateApiKey(prefix = 'sk') {
  const raw = randomBytes(32).toString('base64url');
  return `${prefix}_${raw}`;
}

generateApiKey(); // → "sk_A3f2B1c8D4e9F0a1B2c3..."
generateApiKey('pk'); // → public/publishable key

The entropy comes entirely from randomBytes(32) — that's 256 bits. The prefix and encoding are cosmetic. Some services use UUIDs as API keys; this works but only provides ~122 bits of entropy (version 4 UUID). Use 32+ random bytes for new systems.

The Biggest Mistake: Committing Keys to Git

Every month, security researchers find thousands of valid API keys accidentally committed to public GitHub repositories. GitHub itself scans for common key formats and notifies service providers — but this only works if the provider has registered their pattern. And once a key is in git history, it's there forever even if you delete the file.

If you accidentally commit an API key, rotate it immediately. Don't just delete the file or overwrite it. Assume the key is already compromised — bots scan GitHub within seconds of a push.

# Wrong — hardcoded key in source
OPENAI_API_KEY="sk-abc123..."  # ❌ never do this

# Right — reference via environment variable
OPENAI_API_KEY=os.environ["OPENAI_API_KEY"]  # ✓

# Even better — use a secrets manager
# AWS: boto3 secrets manager
# GCP: Secret Manager API
# HashiCorp Vault, Doppler, Infisical

Storing API Keys on the Server Side

When your service issues API keys to customers, you face a storage dilemma. Unlike passwords, you have two options:

Option A: Store only a hash (like passwords)

Hash the full key with SHA-256 (no need for bcrypt — API keys are already high-entropy) and store only the hash. The key is shown once to the user at creation time. Future authentication re-hashes the presented key and compares. Downside: users can't recover a lost key; they must rotate it.

Option B: Store prefix + hash

Store the first 8 characters in plaintext for user identification purposes (e.g., so you can show "sk_A3f2..."), plus the hash of the full key. This is the Stripe pattern and balances usability with security.

// Creating a new API key — server side
const { randomBytes, createHash } = require('crypto');

function createApiKey() {
  const raw = randomBytes(32).toString('base64url');
  const key = `sk_${raw}`;
  const hash = createHash('sha256').update(key).digest('hex');
  const prefix = key.slice(0, 10); // "sk_A3f2B1c8"

  // Store prefix + hash in DB; return full key once to user
  db.store({ prefix, hash });
  return key; // shown to user exactly once
}

Rotation and Scoping

API keys should be rotated regularly, not just when compromised. Recommended practices:

  • Rotation period: 90 days for production keys; immediately after any team member leaves
  • Minimal scope: Issue separate read-only keys for monitoring, write keys for deployment — don't use one key for everything
  • IP allowlisting: If your service supports it, restrict keys to your server's IPs
  • Key expiration: Set TTLs on keys where possible; short-lived tokens are safer than long-lived keys
  • Audit logging: Log every API key usage with timestamp and IP; set up alerts on anomalous usage

Zero-trust approach: Use short-lived tokens (OAuth 2.0 client credentials flow) instead of long-lived API keys for high-value services. A token that expires in 1 hour does far less damage if compromised than a key valid for 90 days.

Quick Reference: Which to Use When

  • Human logs in to a web app → Password (hashed with Argon2) + MFA
  • Your backend calls a third-party API → API key in secrets manager, injected via env
  • Your SPA calls your own backend → Short-lived JWT (not an API key)
  • CI/CD pipeline deploys your app → API key with minimal deploy scope
  • Webhook signature verification → HMAC secret (technically an API key variant)
