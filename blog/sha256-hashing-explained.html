SHA-256 Explained: How Cryptographic Hashing Works

April 2026 8 min read By Zecure Team

SHA-256 is one of the most widely deployed cryptographic algorithms on the internet — used in TLS certificates, Bitcoin, Git, and hundreds of security protocols. Understanding how it works helps you use it correctly and avoid the common mistake of using it where it doesn't belong.

What Is SHA-256?

SHA-256 (Secure Hash Algorithm 256-bit) is a member of the SHA-2 family of cryptographic hash functions, published by the NSA and standardised by NIST in 2001. It takes any input — a single character, a terabyte file, or anything in between — and produces a fixed-length 256-bit (32-byte) output, represented as 64 hexadecimal characters.

SHA-256 output examples
Input: "hello"
2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
Input: "Hello" (capital H)
185f8db32921bd46d35cc2bc4b4d32d1a4568b07e3e2bc0b6c1e3804dda66a84
Input: "" (empty string)
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855

Notice that changing a single character (lowercase vs uppercase H) produces a completely different hash. This is called the avalanche effect — a core property of good hash functions.

The Four Properties of a Cryptographic Hash Function

SHA-256 (and other cryptographic hashes) must satisfy four fundamental properties:

  1. Deterministic — The same input always produces the same output. Hash("hello") is always the same 64-character string.
  2. Pre-image resistant — Given a hash, it is computationally infeasible to find the original input. You cannot "reverse" a SHA-256 hash.
  3. Second pre-image resistant — Given an input and its hash, it is infeasible to find a different input with the same hash.
  4. Collision resistant — It is infeasible to find any two different inputs that produce the same hash output.

How SHA-256 Works (Simplified)

SHA-256 processes input in 512-bit (64-byte) blocks. Here's a high-level overview of the algorithm:

  1. Padding — The input is padded to a multiple of 512 bits. A 1 bit is appended, followed by zeros, then the original message length as a 64-bit big-endian integer.
  2. Initial hash values — Eight 32-bit words are initialised with the fractional parts of the square roots of the first 8 prime numbers. These constants are publicly known and not secret.
  3. Compression function — Each 512-bit block is processed through 64 rounds of mixing operations involving bitwise AND, OR, XOR, rotations, and additions with 64 round constants (derived from the cube roots of the first 64 primes).
  4. Output — After processing all blocks, the eight 32-bit hash values are concatenated to form the 256-bit digest.

The entire process is designed so that each output bit depends on every input bit, making it impossible to predict or reverse.

SHA-512 is actually faster than SHA-256 on 64-bit processors because it performs fewer rounds on larger blocks. For most applications, SHA-256 provides more than adequate security.

Where SHA-256 Is Used in Practice

  • TLS/HTTPS certificates — Digital signatures in X.509 certificates use SHA-256 (or SHA-384) to sign the certificate content.
  • Git — Git uses SHA-1 historically, but is transitioning to SHA-256 for commit and blob identifiers.
  • Bitcoin — Bitcoin's proof-of-work mining uses SHA-256 twice (SHA-256d) to hash block headers.
  • HMAC-SHA256 — JWT signatures (HS256) and many API authentication schemes use HMAC with SHA-256 to sign messages.
  • API key storage — Instead of storing API keys in plaintext, applications store their SHA-256 hash. On each request, the incoming key is hashed and compared.
  • File integrity — Software publishers distribute SHA-256 checksums alongside downloads so users can verify files haven't been tampered with.

SHA-256 for API Keys and Tokens

When building an API, you should never store API keys in plaintext. Instead, store a SHA-256 hash of each key:

// Node.js: storing and verifying an API key
import crypto from 'crypto';

// When issuing a new key (show raw key to user once)
const rawKey = crypto.randomUUID();
const storedHash = crypto.createHash('sha256').update(rawKey).digest('hex');
// Store storedHash in database, show rawKey to user

// When verifying an incoming request
function verifyKey(incoming, storedHash) {
  const hash = crypto.createHash('sha256').update(incoming).digest('hex');
  // Use timingSafeEqual to prevent timing attacks
  return crypto.timingSafeEqual(Buffer.from(hash), Buffer.from(storedHash));
}

Never use SHA-256 for password storage. SHA-256 can compute ~10 billion hashes per second on a modern GPU. An attacker who steals your database can crack common passwords in seconds. Use bcrypt, Argon2id, or scrypt — these are specifically designed to be slow and memory-intensive.

The Critical Distinction: Hashing vs Password Hashing

This is the most important thing to take away from this article. SHA-256 is a general-purpose cryptographic hash function. It is designed to be fast. Speed is a feature for data integrity, HMAC, and signing. But for password storage, speed is the enemy — an attacker can test billions of guesses per second.

Password hashing algorithms like bcrypt, Argon2id, and scrypt are deliberately slow and memory-intensive. They are designed to remain computationally expensive even as hardware improves, by increasing the cost factor.

Use SHA-256 for: file integrity, API key storage, HMAC signing, digital signatures, data deduplication.
Use bcrypt/Argon2id for: password storage.
