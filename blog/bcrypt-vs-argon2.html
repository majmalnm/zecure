bcrypt vs Argon2: Which Password Hash Should You Use?

April 2025 10 min read By Zecure Team

bcrypt and Argon2id are the two algorithms most developers encounter when building authentication systems. They're both designed to make brute-force attacks expensive — but they take very different approaches, and the choice matters for the security of your users' passwords.

Why Password Hashing Is Not Like Regular Hashing

When you store a password, you should never store it in plaintext — or even as a regular cryptographic hash like SHA-256. Regular hashes are designed to be fast to compute. An attacker with a stolen database of SHA-256 hashes can try billions of guesses per second on a GPU.

Password hashing is different: it's intentionally slow and resource-intensive, making brute-force attacks computationally impractical. Both bcrypt and Argon2 are password hashing functions — but they differ in how they achieve this.

How bcrypt Works

bcrypt was published in 1999 and is based on the Blowfish cipher. It introduced the concept of a configurable cost factor (originally called " rounds") that determines how slow the hash is to compute. Each increment of the cost factor doubles the computation time.

A bcrypt hash looks like this:

$2b$12$LQv3c1yqBWVHxkd0LHAkCOYz6TtxMQJqhN8/X4.S0PKrHqhPWHfeG

The $2b$ denotes the version, $12$ is the cost factor, and the rest is the 184-bit hash output.

bcrypt has one significant limitation: it uses only ~4 KB of memory internally. This makes it relatively cheap to attack with ASICs or FPGAs, because the memory requirement is low enough that an attacker can parallelise massively — running thousands of guesses simultaneously on specialised hardware.

How Argon2 Works

Argon2 was the winner of the Password Hashing Competition in 2015 and comes in three variants — Argon2d, Argon2i, and Argon2id. For password storage, you want Argon2id, which blends both approaches: Argon2d (data-dependent, resistant to GPU/ASIC attacks) and Argon2i (data-independent, resistant to side-channel attacks).

Argon2id has three tunable parameters:

  • Memory cost (m) — How much RAM to use per hash (e.g., 64 MB)
  • Time cost (t) — Number of iterations to perform
  • Parallelism (p) — Number of parallel threads

The key advantage of Argon2 over bcrypt is that it requires a configurable amount of memory per hash — making it expensive to parallelise on ASICs and GPUs, which have limited memory bandwidth per compute unit.

Real-World Attack Costs

Here's what the difference means in practice. A high-end GPU (NVIDIA H100) can compute approximately:

  • bcrypt (cost 12): ~70,000 guesses/second
  • Argon2id (64 MB, 3 iterations): ~500 guesses/second

The attacker with Argon2id gets 140× fewer guesses per second per GPU — and that's before accounting for the fact that the GPU's memory bandwidth limits how many can run in parallel. For a 6-character lowercase password (266 ≈ 309 million combinations), bcrypt at cost 12 would take ~74 minutes to exhaust; Argon2id would take ~7 days. For longer passwords, the gap widens dramatically.

PBKDF2 (e.g., Django's default) is still widely used but has the same fundamental weakness as bcrypt — it's fast to compute and easy to parallelise on GPUs. If you're choosing a new algorithm, bcrypt or Argon2id are both strictly better.

Which Should You Use?

The answer depends on your situation:

Use Argon2id if...

  • You're building a new system and can make a fresh choice
  • Security overhead of Argon2id (memory/CPU cost) is acceptable for your application
  • You're using a modern framework with built-in Argon2 support (PHP 7.2+, Python's argon2-cffi, Go's cryptoargon2)
  • You want the best available protection against GPU-accelerated brute-force attacks

Use bcrypt if...

  • You're on an older framework where Argon2 support is lacking or immature
  • You're maintaining an existing system where bcrypt is already in use — migration costs often outweigh the security benefit
  • Your deployment environment has strict memory constraints (Argon2id with 64 MB per hash may not suit serverless)
  • You need maximum compatibility across languages and runtimes

If you're starting fresh and your stack supports it — use Argon2id. If you're using bcrypt, don't panic — it's still secure when configured with a high enough cost factor (12 or above on modern hardware). The most important thing is not to use a fast hash like SHA-256 or MD5.

Tuning Parameters

Both algorithms require tuning to match your security needs and hardware constraints. The goal is to make hashing as slow as tolerable for legitimate users while making attacks prohibitively expensive.

bcrypt tuning

The only tunable parameter is the cost factor. The right cost factor depends on your server hardware — aim for a hashing time of 200–500ms. On a modern server, cost 12 takes ~250ms; cost 13 takes ~500ms. Recalibrate as hardware improves.

Argon2id tuning

The OWASP 2023 guidelines recommend:

  • Memory: 64 MB minimum (19 MB minimum)
  • Iterations: 3 minimum
  • Parallelism: 1 or greater

For higher-security applications, use 128 MB or more. The Argon2 documentation recommends calibrating so that a single hash takes roughly 1 second on your target hardware.

What About Existing Systems?

If you're currently using bcrypt or PBKDF2, you don't necessarily need to migrate immediately. The key question is whether your cost/tuning parameters are adequate for modern hardware. Run a benchmark — if a single hash takes less than 100ms, your parameters are too low.

When you do migrate, the standard approach is to re-hash on login: when a user next successfully authenticates, verify their password against the old hash, then generate a new hash with the updated algorithm and store it. This avoids forced password resets.

Summary

Argon2id is the stronger algorithm — its memory-hard design makes GPU and ASIC attacks dramatically more expensive. bcrypt is still secure with proper tuning, but its low memory cost means a determined attacker with commodity hardware can still make many guesses per second.

For new projects, Argon2id is the clear choice. For existing projects, upgrading your cost factors on bcrypt is a pragmatic intermediate step that delivers meaningful security improvement without a full migration. Whichever you use, ensure your parameters are calibrated to modern hardware — a hash that completes in milliseconds is not providing adequate protection.
