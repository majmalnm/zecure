How to Safely Store Environment Variables and Secrets

August 2025 8 min read By Zecure Team

GitHub scans public repos for leaked secrets and has found millions. A single committed .env file can expose your database, your users' data, and your cloud bill. Here's a developer's guide to doing this right — from local development to production.

The Problem: Secrets End Up in Git

The most common way API keys leak is simple: a developer puts credentials in a .env file, forgets to add .env to .gitignore, and pushes to GitHub. Or they hardcode a secret directly in source code during a quick test and never remove it. Or they use environment variables in a CI/CD config file that gets committed.

Once a secret appears in a git commit — even if you delete it immediately — it lives in the repository history. Anyone who clones the repo can run git log and find it. Bots scrape GitHub continuously for API keys, database credentials, and private keys.

If you accidentally committed a secret: Assume it's compromised. Rotate the key immediately — don't wait to check if anyone found it. Bots scan GitHub in minutes. Rewriting git history is not a substitute for rotating credentials.

The Tiered Approach to Secrets Storage

There's no single solution for every situation. The right tool depends on where the secret lives and who needs to access it:

Never
Hardcoded in source
Secrets in .js, .py, or config files committed to version control. Immediately compromised.
Dev only
.env file (gitignored)
Fine for local development. Never in production. Add .env to .gitignore before the first commit.
Acceptable
Platform env vars
Vercel, Railway, Render, Heroku all have secret env var storage. Better than .env, but audit access carefully.
Production
Secrets manager
AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault. Audited, rotatable, access-controlled.

The Right .env Setup for Local Development

For local development, .env files are the standard. The rules are simple:

# .gitignore — add this before your first commit
.env
.env.local
.env.*.local

# Commit a template — never actual values
.env.example
# .env.example — committed to git, shows required vars with no real values
DATABASE_URL=postgresql://user:password@host:5432/dbname
STRIPE_SECRET_KEY=sk_live_...
JWT_SECRET=replace-with-64-random-chars
AES_KEY=replace-with-64-hex-chars

Commit .env.example (with dummy values) as documentation. Each developer creates their own local .env with real values. This file never gets committed.

Verify .gitignore before you push

# Check whether git would track your .env file
git check-ignore -v .env

# If it returns nothing, .env is NOT ignored — fix this before committing
echo ".env" >> .gitignore

CI/CD: Where Secrets Usually Break

CI/CD pipelines need secrets to deploy — database URLs, cloud credentials, API keys. The wrong approaches are common:

# WRONG — hardcoded in workflow file
env:
  DATABASE_URL: postgresql://admin:hunter2@prod-db:5432/app

# RIGHT — reference from platform secrets store
env:
  DATABASE_URL: ${{ secrets.DATABASE_URL }}

GitHub Actions, GitLab CI, and CircleCI all have encrypted secret stores. Never put real values in workflow YAML files. Reference secrets by name — the platform injects the value at runtime without it appearing in logs or history.

Watch for echo in CI logs. If you accidentally echo $DATABASE_URL in a CI step, the secret may appear in build logs — which are often world-readable on public repos. Always check log output after adding debugging steps.

Production: Use a Secrets Manager

For production workloads, environment variables injected by a platform are acceptable for simple deployments, but a dedicated secrets manager is the right answer for anything that matters:

AWS Secrets Manager

// Node.js — fetch secret at runtime, not at deploy time
import { SecretsManagerClient, GetSecretValueCommand } from '@aws-sdk/client-secrets-manager';

const client = new SecretsManagerClient({ region: 'us-east-1' });

const secret = await client.send(new GetSecretValueCommand({
  SecretId: 'prod/myapp/database'
}));
const { DB_PASSWORD } = JSON.parse(secret.SecretString);

HashiCorp Vault

# Vault CLI — read a secret
vault kv get secret/myapp/prod

# Application fetches secrets at startup via Vault API
# Access is controlled by Vault policies tied to app identity

The key advantages of a secrets manager over platform env vars: auditing (who accessed what, when), rotation (update the secret in one place, all apps get it), and access control (apps only get the secrets they need, not everything).

Generating Secrets Worth Protecting

A secrets manager is useless if the secrets themselves are weak. Every secret in your infrastructure — database passwords, JWT signing keys, encryption keys — should be:

  • Generated by a CSPRNG, never typed by a human
  • At least 128 bits of entropy (32+ random characters from a large set)
  • Unique per environment (dev, staging, prod each get different values)
  • Rotated when a team member with access leaves
// Node.js — generate a 256-bit secret for JWT signing or AES encryption
const { randomBytes } = require('crypto');
console.log(randomBytes(32).toString('hex'));
// → 64 hex chars (256 bits). Paste this into your secrets manager.

Checklist

  • .env in .gitignore — verify with git check-ignore -v .env before first push
  • .env.example committed — documents required vars without real values
  • CI secrets in platform store — GitHub Actions secrets, not hardcoded in YAML
  • Production uses a secrets manager — with auditing and rotation
  • Each secret is unique per environment — prod and staging use different values
  • Secrets are randomly generated — not derived from names, dates, or memorable phrases
  • Rotation plan exists — know how to rotate each secret and when
